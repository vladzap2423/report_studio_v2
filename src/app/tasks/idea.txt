Tasks — концепт фичи (с группами)

1. Цель
Сделать страницу /tasks, где сотрудники могут:
- создавать задачи;
- передавать задачи;
- выполнять и закрывать задачи;
- оставлять комментарии;
- видеть историю изменений.

Ключевое правило:
доступ к /tasks есть только у пользователей, которые состоят хотя бы в одной task-группе.

2. Базовая модель доступа
- Фичей управляет роль god.
- God создает группы и назначает пользователей в группы.
- Если пользователь не состоит ни в одной группе, страница /tasks ему недоступна.
- Если пользователь состоит в группе, он видит задачи выбранной группы и может работать только внутри этой группы.
- Пользователь может состоять в нескольких группах.

3. Роли и права
- god:
  - создает/редактирует/архивирует группы;
  - добавляет/удаляет участников групп;
  - может назначать владельцев/модераторов групп (опционально);
  - имеет доступ ко всем группам и всем задачам.
- admin и user (одинаковые права на /tasks):
  - работают только в тех группах, где состоят;
  - создают задачи в текущей группе;
  - назначают задачи себе или участникам текущей группы;
  - меняют статус по разрешенным переходам;
  - могут менять приоритет/срок/исполнителя задач внутри своей группы;
  - могут закрывать/отменять задачи внутри своей группы.

4. Сущности

4.1 task_group
- id
- name (уникальное имя группы)
- description (nullable)
- is_active
- created_by
- created_at
- updated_at

4.2 task_group_member
- group_id
- user_id
- member_role (member, manager) — опционально для расширения
- joined_at

4.3 task
- id
- group_id (обязательная привязка задачи к группе)
- title
- description
- status (new, in_progress, blocked, review, done, canceled)
- priority (low, medium, high)
- creator_id
- assignee_id
- due_at (nullable)
- created_at
- updated_at
- started_at (nullable)
- completed_at (nullable)

4.4 task_comment
- id
- task_id
- author_id
- body
- created_at

4.5 task_history
- id
- task_id
- actor_id
- action (create, update, transfer, status_change, priority_change, comment_add, reassign)
- old_value (jsonb, nullable)
- new_value (jsonb, nullable)
- created_at

4.6 task_watcher (опционально)
- task_id
- user_id

5. Основные ограничения бизнес-логики
- Нельзя создать задачу без group_id.
- Нельзя назначить исполнителя, который не состоит в этой группе.
- Нельзя передать задачу пользователю вне группы.
- Пользователь видит только задачи текущей выбранной группы.
- God может смотреть и редактировать все группы (в том числе в режиме "Все группы").

6. Переключение между группами (важно для UX)

6.1 В шапке /tasks
- Селектор "Текущая группа" (dropdown).
- Быстрый поиск по группам в селекторе.
- Отображение количества задач в выбранной группе.

6.2 Поведение
- Выбор группы фильтрует список задач и исполнителей.
- Последняя выбранная группа сохраняется:
  - в URL: /tasks?groupId=...
  - и в localStorage как fallback.
- Если в URL groupId недоступен пользователю, открывается первая доступная группа.

6.3 Для пользователей с несколькими группами
- Переключение без перезагрузки страницы.
- Отдельные фильтры/сортировки можно помнить для каждой группы.

7. Статусы и переходы
- new -> in_progress
- in_progress -> review
- review -> done
- in_progress -> blocked
- blocked -> in_progress
- new|in_progress|blocked|review -> canceled

Правило:
- первый переход в in_progress фиксирует started_at;
- переход в done фиксирует completed_at.

8. Приоритеты
- high (красный)
- medium (желтый)
- low (бежевый)

9. UI страницы /tasks

9.1 Верхняя панель
- Выбор группы (обязательный контекст).
- Кнопка "Создать задачу".
- Поиск по названию/описанию.
- Фильтры: статус, приоритет, исполнитель, просроченные.
- Переключатель вида: список / канбан.

9.2 Список задач
- Колонки: Приоритет, Название, Статус, Исполнитель, Дедлайн, Обновлено.
- Быстрые действия: взять в работу, передать, закрыть.

9.3 Карточка задачи
- Заголовок, описание, статус, приоритет, дедлайн.
- Исполнитель (только из участников текущей группы).
- История изменений.
- Комментарии.

10. Передача задачи
- Действие "Передать".
- Поля: новый исполнитель (список участников группы) + обязательный комментарий.
- После подтверждения:
  - обновляется assignee_id;
  - в task_history пишется transfer;
  - добавляется системный комментарий о передаче.

11. API (черновой контракт)

11.1 Группы
- GET /api/task-groups (мои группы)
- POST /api/task-groups (god)
- PATCH /api/task-groups/:id (god)
- POST /api/task-groups/:id/members (god)
- DELETE /api/task-groups/:id/members/:userId (god)

11.2 Задачи
- GET /api/tasks?groupId=...
- POST /api/tasks
- GET /api/tasks/:id
- PATCH /api/tasks/:id
- POST /api/tasks/:id/status
- POST /api/tasks/:id/transfer
- GET /api/tasks/:id/comments
- POST /api/tasks/:id/comments
- GET /api/tasks/:id/history

12. БД (минимальный набор таблиц)
- task_groups
- task_group_members
- tasks
- task_comments
- task_history

Индексы:
- task_group_members(user_id, group_id)
- tasks(group_id, status, priority, due_at)
- tasks(group_id, assignee_id, status)
- task_comments(task_id, created_at)
- task_history(task_id, created_at)

13. MVP (первый релиз)
1. Таблицы: task_groups, task_group_members, tasks, task_comments, task_history.
2. Ограничение доступа к /tasks: только пользователи с группами.
3. Переключатель групп на странице.
4. Список задач в контексте выбранной группы.
5. Создание задачи и назначение только на участников группы.
6. Изменение статуса и передача задачи внутри группы.
7. Комментарии и история изменений.

14. После MVP
- Канбан доска по выбранной группе.
- Роли внутри группы (manager/member).
- Подписчики, уведомления, SLA и отчеты.
- Вложения к задачам.

15. Критерии готовности
- Пользователь без групп не имеет доступа к /tasks.
- Пользователь с несколькими группами может быстро переключаться между ними.
- Нельзя назначить/передать задачу пользователю вне группы.
- Все изменения задачи пишутся в history.
- Поиск и фильтры работают в рамках текущей группы.
